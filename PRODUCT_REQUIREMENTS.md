# Metis 紫微斗数产品整改需求文档

## 1. 项目目标

### 1.1 一句话定位

Metis 是一个面向普通用户的 AI 紫微斗数命盘产品，核心价值是让用户在 3 分钟内生成一份看得懂、愿意继续追问、愿意分享的命盘解读。

### 1.2 当前问题

当前项目已经具备基础排盘、AI 解读、合盘、知识库、古籍库等模块，但整体仍停留在 MVP Demo 阶段，主要问题包括：

- 首页和部分页面存在无功能按钮，用户会误以为产品坏了。
- 多个页面是展示型占位页，缺少可操作闭环。
- 中文文案疑似存在编码或内容损坏问题，影响专业感和信任感。
- AI 解读虽然可用，但缺少稳定的结构、失败兜底、保存、复制、分享等产品能力。
- 功能入口较多，但核心主路径不够聚焦。
- 专业版、登录、会员等商业化入口存在，但没有形成完整转化逻辑。

### 1.3 第一阶段目标

第一阶段不追求做完整命理平台，而是先完成一个可上线、可传播、可验证商业价值的核心产品闭环：

```text
首页进入
  -> 输入出生信息
  -> 生成紫微命盘
  -> 获得结构化 AI 解读
  -> 支持继续追问
  -> 支持复制 / 分享
  -> 引导专业版或等待名单
```

### 1.4 成功标准

第一阶段完成后，产品应达到：

- 所有可见按钮都有明确结果，不存在“点了没反应”的假按钮。
- 用户可以在移动端和桌面端顺利完成一次排盘。
- 命盘结果页默认生成一份可读的命格总览。
- 用户可以切换感情、事业、财运、健康、性格等主题继续生成解读。
- AI 失败时有明确提示和本地兜底内容，不出现空白结果。
- 合盘功能可以完成双方信息输入、生成分析、继续追问。
- 登录 / 专业版入口不再是尴尬占位，而是可收集线索或清楚说明权益。
- 页面文案统一、可信、不过度承诺。

## 2. 用户与使用场景

### 2.1 目标用户

#### 普通小白用户

特点：

- 对紫微斗数感兴趣，但看不懂传统术语。
- 关心事业、感情、财运、未来一年。
- 希望结果直接、好懂、有建议。

产品重点：

- 降低理解门槛。
- 少术语，多解释。
- 多给结构化结论和可执行建议。

#### 命理兴趣用户

特点：

- 对主星、宫位、四化、大限、流年有一定了解。
- 希望看到更专业、更细的分析。
- 可能愿意为深度报告付费。

产品重点：

- 提供宫位、三方四正、四化解释。
- 保留传统命理信息。
- 提供深度专题报告入口。

#### 潜在付费用户

特点：

- 希望保存命盘。
- 希望导出报告。
- 希望针对感情、事业、流年获得深度解读。
- 可能需要合盘分析。

产品重点：

- 清楚展示免费版和专业版差异。
- 在用户已经看到价值后再引导付费。
- 不做强迫式转化。

### 2.2 核心场景

#### 场景 A：第一次排盘

用户从首页点击“立即排盘”，输入出生日期、时间、地点、性别，生成命盘，并自动获得命格总览。

#### 场景 B：看专项解读

用户在命盘结果页点击“感情”“事业”“财运”“健康”“性格”等主题，AI 生成对应维度的结构化解读。

#### 场景 C：继续追问

用户看完解读后，可以输入具体问题，例如：

- 我今年适合换工作吗？
- 我的感情问题主要在哪里？
- 我适合创业还是上班？
- 未来一年需要注意什么？

#### 场景 D：合盘分析

用户输入两个人的出生信息，选择关系类型，生成合盘分析，并支持继续追问。

#### 场景 E：分享结果

用户可以复制解读内容，或生成一张简洁好看的分享卡片。

#### 场景 F：专业版转化

用户在完成排盘或合盘后，看到专业版权益，例如：

- 保存历史命盘
- 导出完整报告
- 深度流年分析
- 合盘深度报告
- 无限追问

第一阶段可以不做真实支付，先做等待名单或联系方式收集。

## 4. 页面需求

### 4.1 首页

#### 目标

首页只承担一个核心任务：让用户快速理解产品价值，并进入排盘或合盘。

#### 主要改动

- 明确主 CTA：立即排盘。
- 次 CTA：合盘分析。
- 首页功能模块保留，但不要让用户迷失。
- 所有顶部按钮必须可用或移除。

#### 按钮规则

| 元素 | 处理方式 |
|---|---|
| 中 | 改成当前语言状态，不作为按钮 |
| EN | 第一阶段隐藏，避免假按钮 |
| 专业版 | 跳转到专业版 / 等待名单页 |
| 菜单 | 移动端展开导航；桌面端不显示 |
| 排盘模块 | 跳转 `/chart` |
| 合盘模块 | 跳转 `/heming` |
| 命理双子模块 | 跳转 `/twins`，页面必须说明当前能力 |
| 天纪 / 地纪 / 人纪 | 跳转对应内容页，不做假交互 |

#### 验收标准

- 首页无点击无反应按钮。
- 首屏能明确看到产品名称和核心价值。
- 用户 5 秒内能找到“立即排盘”入口。
- 移动端导航可用。

### 4.2 排盘页 `/chart`

#### 目标

这是第一阶段最核心页面。用户必须能顺畅完成排盘，并获得高质量解读。

#### 表单需求

- 字段包括：姓名、出生日期、出生时间、是否不知道出生时间、出生地点、性别。
- 出生地点默认可以选择北京，减少用户输入成本。
- 农历功能第一阶段不开放时，不要做 disabled 按钮；改为“农历即将支持”的轻提示。
- 表单错误提示必须是正常中文。
- 提交中显示清楚 loading。

#### 命盘结果需求

- 生成命盘后自动滚动到结果区。
- 左侧展示命盘。
- 右侧展示 AI 解读。
- 顶部展示命主摘要：日期、地点、性别、真太阳时或时辰。
- 提供“重新排盘”。
- 提供“复制解读”。
- 提供“分享命盘”。

#### AI 主题需求

默认主题：

- 命格总览

可切换主题：

- 感情
- 事业
- 财运
- 健康
- 性格

每个主题输出结构：

```text
一句话结论
核心配置
优势
风险
现实建议
未来一年重点
```

#### AI 失败兜底

当远程 AI 不可用时：

- 显示“AI 暂时不可用，已为你生成基础本地解读”。
- 使用本地规则生成基础解读。
- 不允许结果区域空白。

#### 验收标准

- 完整输入后可以生成命盘。
- 默认自动生成命格总览。
- 点击主题按钮可以生成对应主题。
- AI 失败时仍有本地内容。
- 复制按钮可复制当前解读。
- 重新排盘可回到表单。

### 4.3 合盘页 `/heming`

#### 目标

让用户输入两个人的信息，生成关系分析，并能继续追问。

#### 表单需求

- A 方和 B 方分别填写出生信息。
- 支持关系类型选择：
  - 恋人 / 伴侣
  - 合伙 / 同事
  - 朋友
  - 亲子 / 家人
- 未填写完整时，点击分析需要明确提示缺失项。

#### 结果需求

合盘分析结构：

```text
整体匹配度
关系关键词
双方吸引点
主要冲突点
相处建议
未来需要注意的阶段
```

#### 继续追问

默认快捷问题：

- 两个人适合长期发展吗？
- 最大冲突点是什么？
- 谁更容易主动？
- 财运是否互补？
- 适合一起创业吗？

#### 验收标准

- 双方信息完整后可以生成合盘分析。
- 分析中有 loading 状态。
- 分析失败有清楚提示。
- 快捷问题可继续生成回答。
- 自定义问题可提交。

### 4.4 登录 / 专业版页 `/login`

#### 目标

第一阶段不做真实登录，改成有效的商业转化页。

#### 页面内容

- 标题：专业版即将开放。
- 说明免费版已支持基础排盘和合盘。
- 展示专业版权益：
  - 保存历史命盘
  - 深度命盘报告
  - 流年分析
  - 合盘深度报告
  - 导出报告
  - 无限追问
- 提供等待名单入口：
  - 邮箱输入
  - 微信号输入
  - 提交按钮

第一阶段如果没有后端，可以先保存到 localStorage，或用 mailto / 表单服务替代。

#### 验收标准

- 页面不再像占位说明。
- 用户知道专业版有什么价值。
- 用户可以留下联系方式或看到明确行动入口。

### 4.5 命理双子页 `/twins`

#### 目标

第一阶段不做真实数据库匹配，但不能让页面显得无效。

#### 页面内容

- 解释“命理双子”概念。
- 引导用户先排盘。
- 展示未来功能：
  - 找到相同命盘结构的人
  - 对比不同人生选择
  - 地域与职业样本分析
- 提供 CTA：先生成我的命盘。

#### 验收标准

- 页面不出现假搜索、假匹配。
- CTA 跳转 `/chart`。
- 用户理解这是后续功能。

### 4.6 天纪 / 地纪 / 人纪页面

#### 目标

第一阶段作为内容索引页，不做复杂交互。

#### 页面要求

- 每个模块卡片如果不能点击，就不要表现成按钮。
- 如果可以点击，需要进入对应详情或知识库。
- 页面文案必须正常中文。
- 保留图片和简洁说明。

#### 验收标准

- 不存在看起来能点但没反应的卡片。
- 页面能作为 SEO 内容入口。

## 6. AI 解读需求

### 6.1 基本原则

AI 解读必须：

- 说人话。
- 有结构。
- 有结论。
- 有建议。
- 避免吓人。
- 避免绝对化断言。
- 明确是传统文化参考。

### 6.2 禁止输出风格

避免：

- “你一定会……”
- “必然破财……”
- “注定失败……”
- “必须付费才能化解……”
- 过度恐吓。
- 过度玄学化。

### 6.3 推荐输出结构

每次 AI 解读尽量使用以下结构：

```text
一句话结论
命盘依据
优势
风险
现实建议
适合行动
需要避免
```

### 6.4 追问能力

用户追问时，AI 应基于当前命盘回答，不重新泛泛解释。

示例：

用户问：“我今年适合换工作吗？”

回答应包含：

- 趋势判断
- 适合换的条件
- 不适合换的风险
- 更稳妥的行动建议

### 6.5 免责声明

所有 AI 解读区域底部需要有轻量提示：

```text
以上内容基于传统紫微斗数文化模型生成，仅供自我理解与娱乐参考，不构成现实决策、医疗、法律或投资建议。
```

## 9. 技术要求

### 9.1 技术原则

- 最小改动。
- 不做大重构。
- 优先修主路径。
- 优先复用现有组件。
- 不引入不必要依赖。
- 保持 Next.js / React / TypeScript / Tailwind 体系。

### 9.2 编码问题

必须确认源码和页面中文是否正常。

如果浏览器页面也乱码：

- 优先修复所有用户可见中文。
- 确保文件保存为 UTF-8。
- 修复 AI prompt 中损坏的中文。
- 修复 SEO metadata 中损坏的中文。

如果只是终端显示乱码：

- 记录原因。
- 不做无意义大规模替换。

### 9.3 构建问题

当前构建存在多 lockfile 警告。

处理建议：

- 保留项目目录 `ziwei-doushu-latest/package-lock.json`。
- 删除或忽略根目录无意义 `package-lock.json`。
- 或在 `next.config.js` 配置 `outputFileTracingRoot`。

第一阶段只需要消除明显部署风险，不做额外工程化改造。

任务：

- 每日 / 每月提醒。
- SEO 内容体系。
- 命理双子样本库。

## 12. 默认决策

为了避免开发时反复摇摆，第一阶段默认采用以下决策：

- 不做真实登录，先做专业版等待名单。
- 不做真实支付，先验证用户兴趣。
- 不做英文版，先隐藏英文按钮。
- 不做农历转换，先以公历为主。
- 不做命理双子数据库，先作为概念介绍和排盘引导。
- 不大改技术架构，优先修体验和主路径。
- 不追求一次做到 100 分，先做到 80 分可上线。
