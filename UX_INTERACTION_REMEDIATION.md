# Metis 全局交互体验整改报告与执行方案

日期：2026-07-13
项目：`ziwei-doushu-latest`
目标：让新用户在桌面端和移动端都能清楚找到入口、完成排盘、读懂结果并继续操作。

## 一、真实用户模拟

本轮模拟用户：第一次访问 Metis、不了解紫微斗数术语、使用桌面端 1440×900 和移动端 390×844。

模拟任务：

1. 从首页找到“立即排盘”。
2. 填写出生日期、时间、地点和性别。
3. 生成命盘并确认结果已经出现。
4. 切换到“事业”主题并继续追问。
5. 进入合盘页面，确认双方表单可填写。
6. 在移动端完成以上入口识别，不发生横向滚动。

证据截图保存在：`audit/2026-07-13-global-ux/`

## 二、走查发现

### P0：阻塞完成任务

| 编号 | 问题 | 证据 | 影响 |
|---|---|---|---|
| P0-1 | 移动端顶部导航隐藏所有功能入口，只保留“中 / EN” | `07-mobile-home.png` | 新用户不知道如何直接进入起盘或合盘 |
| P0-2 | 首页首屏没有明确的“立即排盘”文案 CTA，主要依赖图片卡片 | `01-home.png` | 用户需要猜测图片可以点击，首屏转化路径不明确 |
| P0-3 | 表单提交按钮在移动端首屏不可见 | `02-chart-form.png` | 用户填写后还要长距离滚动，容易误以为没有提交入口 |

### P1：明显降低信任和理解成本

| 编号 | 问题 | 证据 | 影响 |
|---|---|---|---|
| P1-1 | “农历”呈现为 disabled 按钮，像是坏掉的功能 | `02-chart-form.png` | 用户会产生“按钮不能用”的负面反馈 |
| P1-2 | 命盘结果区信息密度高，主题切换缺少可访问语义 | `04-chart-result.png` | 用户难以知道当前主题和生成状态 |
| P1-3 | 首页模块文案偏研究型，第一屏对普通用户的任务指向不够直接 | `01-home.png` | 价值理解依赖阅读和探索，首次转化较慢 |

### P2：体验与维护问题

| 编号 | 问题 | 证据 | 处理方式 |
|---|---|---|---|
| P2-1 | 开发环境存在 Next.js 1 Issue 浮层 | `04-chart-result.png` | 生产环境复验，避免把开发提示误判为产品 UI |
| P2-2 | 控件状态需要统一键盘和屏幕阅读器语义 | DOM 走查 | 增加 `aria-current`、`aria-pressed`、`aria-live` |
| P2-3 | 部分功能是研究入口，普通用户不容易区分“现在可用”和“即将支持” | 首页、表单 | 文案明确化，不保留假按钮 |

## 三、整改原则

1. 每个可点击元素必须有明确结果。
2. 首屏只保留两条核心路径：立即排盘、合盘分析。
3. 不可用能力不伪装成可点击按钮。
4. 结果区必须告诉用户当前主题、是否正在生成、下一步可以做什么。
5. 移动端优先保证表单提交和结果阅读。
6. 本地知识引擎输出必须进入现有结果框，不用空白占位。

## 四、本轮执行清单

- [x] 移动端显示“起盘 / 合盘”快捷入口。
- [x] 首页首屏卡片增加明确进入 CTA。
- [x] 移动端表单提交按钮保持可见。
- [x] 农历改为“即将支持”状态文本，不再使用死按钮。

- [x] 继续使用本地知识引擎，不调用大模型。

## 五、验收标准

### 首页

- 桌面端 5 秒内能看到起盘入口。
- 移动端首屏能看到“起盘”和“合盘”。
- 首屏卡片有明确进入动作。

### 表单

- 日期、时间、地点、性别填写后，移动端无需滚回页面底部即可提交。
- 农历状态明确说明“即将支持”。
- 空表单提交显示中文错误，不出现无反馈。

### 结果

- 命盘生成后显示命盘、结构化解读和当前主题。
- 主题切换后显示对应主题内容或明确 loading 状态。
- 追问、复制和分享入口仍然可用。

### 合盘

- 双方表单可独立填写。
- 缺少信息时显示明确错误。
- 完整填写后输出本地知识库合盘结果。

### 工程

- `npm.cmd run build` 通过。
- 3001 端口开发服务可访问。
- 浏览器桌面端和移动端无横向溢出。
- 无新的控制台错误。
